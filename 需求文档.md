# 网页版视频对话软件需求文档

## 1. 项目概述

开发一个基于WebRTC的网页版视频对话软件，实现P2P视频通话，不通过服务器转发音视频数据，仅通过服务器转发房间号和信令信息。项目适合原生部署在Ubuntu服务器上，未来可逐步扩展功能。

## 2. 核心功能

### 2.1 房间管理
- **房间创建**：用户点击通话按钮生成唯一房间号（如6-8位数字）
- **房间加入**：用户通过填写房间号加入已创建的房间
- **房间限制**：每个房间最多支持2位用户，超过2位时拒绝后续用户加入
- **房间销毁**：当房间内用户全部离开时，自动销毁房间

### 2.2 视频通话
- **P2P通信**：音视频数据直接在用户浏览器之间传输，不通过服务器转发
- **实时连接**：建立低延迟的视频通话连接
- **连接状态**：显示视频通话连接状态（连接中、已连接、断开）

### 2.3 服务器功能
- **房间号转发**：仅通过服务器转发房间号信息
- **信令交换**：转发WebRTC连接所需的信令（如ICE候选者、SDP描述）
- **状态管理**：维护房间状态和用户信息

## 3. 技术需求

### 3.1 前端技术
- **框架**：React + TypeScript
- **通信**：WebRTC API（实现P2P连接）
- **实时通信**：Socket.io Client（与服务器交换信令）
- **UI**：Tailwind CSS（构建现代界面）

### 3.2 后端技术
- **运行环境**：Node.js 14+
- **框架**：Express（轻量级服务器）
- **实时通信**：Socket.io Server（处理WebSocket连接）
- **状态存储**：内存存储（或Redis，用于生产环境）

### 3.3 部署环境
- **服务器**：Ubuntu
- **反向代理**：Nginx（处理HTTPS请求）
- **进程管理**：PM2（确保服务稳定运行）
- **网络**：HTTPS协议（确保信令安全）

## 4. 技术实现细节

### 4.1 WebRTC实现
- **信令流程**：
  1. 用户A创建房间，生成房间号
  2. 用户B通过房间号加入房间
  3. 服务器将用户B的加入请求转发给用户A
  4. 用户A和B通过服务器交换SDP描述和ICE候选者
  5. 建立P2P连接，开始视频通话

- **NAT穿透**：使用STUN服务器（默认Google STUN），必要时配置TURN服务器作为备选

### 4.2 房间管理逻辑
- **房间号生成**：使用随机数生成器生成唯一的6-8位数字房间号
- **房间状态**：每个房间包含创建者信息、当前用户列表、创建时间等
- **用户匹配**：当第二位用户加入时，服务器通知两位用户开始信令交换
- **用户离开**：当用户断开连接时，服务器更新房间状态，若房间为空则销毁

### 4.3 安全措施
- **HTTPS**：使用HTTPS协议保护信令交换
- **房间访问**：验证房间号有效性，防止未授权访问
- **WebRTC安全**：配置安全的ICE服务器，防止恶意攻击

## 5. 开发规范

### 5.1 代码结构
- **前端**：组件化开发，分离UI组件和业务逻辑
  - `src/components/`：UI组件
  - `src/services/`：WebRTC和Socket.io服务
  - `src/hooks/`：自定义hooks
- **后端**：模块化设计，清晰的路由和服务层
  - `server/routes/`：API路由
  - `server/services/`：业务逻辑
  - `server/models/`：数据模型

### 5.2 测试策略
- **单元测试**：测试核心功能模块
- **集成测试**：测试前后端交互
- **端到端测试**：测试完整的通话流程
- **网络测试**：在不同网络环境下测试连接稳定性

### 5.3 性能优化
- **连接速度**：优化WebRTC连接建立过程，减少连接时间
- **视频质量**：实现自适应视频质量，根据网络状况调整分辨率
- **服务器负载**：使用Socket.io的事件驱动模型，高效处理并发连接

## 6. 部署流程

1. **环境准备**：在Ubuntu服务器上安装Node.js、Nginx、PM2
2. **代码部署**：将前端构建产物和后端代码部署到服务器
3. **配置Nginx**：设置反向代理，配置HTTPS证书
4. **启动服务**：使用PM2启动后端服务，设置自动重启
5. **监控**：配置日志记录和监控，确保服务稳定运行
